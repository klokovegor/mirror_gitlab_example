План лабораторной работы №9 (метод Ньютона и бисекция)

Источник: только файл методички. Общее правило из PDF: после каждой решённой задачи/подзадачи — отдельный коммит с именованием по соответствующим md-файлам и сразу push; без промежуточных коммитов работа может не быть принята.



Этап 1. Первичная реализация методов и тестов





Изучить опорные материалы по ссылкам из задания: метод бисекции, метод Ньютона (раздел C++) — согласовать сигнатуры с википедийными примерами.



Реализовать численные методы максимально простым, читаемым кодом; вместо std::function допускаются лямбды или указатели на функцию (как в первом семестре).



Согласовать целевые сигнатуры с PDF (фрагмент из методички; обратить внимание: у iteration_count в тексте PDF нет значения по умолчанию — при реализации уточнить разумный лимит итераций в коде/ТЗ):

double solve_bisection(..., double epsilon_x = 1e-8, std::size_t iteration_count);
double solve_newton(..., double epsilon_x = 1e-8, std::size_t iteration_count);





Написать тесты на поиск корней для четырёх типов: линейное уравнение, квадратное, гиперболическое, гармоническая функция.



Отладчиком проверить, за сколько итераций сходится расчёт для линейной и квадратичной функции для каждого метода (бисекция и Ньютон).



При использовании ИИ (по методичке): подготовить documents/lab9-numerical/1_basic_tz.md (ТЗ), затем documents/lab9-numerical/1_basic_plan.md (план: какие файлы, что в них), реализовать по плану, коммит с названием 1tz_basic, убедиться что коммит ушёл на сервер.



Этап 2. Дополнительные критерии останова и интерфейс результата





Метод Ньютона — критерии останова: по приращению аргумента Δx, по невязке r(x), по числу итераций n.



Метод бисекции — критерии: по ширине интервала Li, по невязке r(x), по n.



Результат расчёта: не только корень, но и какой критерий остановки сработал; пользователь должен иметь возможность это узнать через интерфейс.



Гибкость: критерии по невязке и по аргументу должны подключаться по отдельности или вместе — продумать минималистичный способ (в PDF указание: struct для параметров алгоритма и для возврата результата).



Подобрать данные, чтобы в тестах реально сработал каждый критерий; реализовать тесты.



Документ для всех: кратко описать принятые решения по интерфейсу солверов и проверке — файл lab9-methods_implementation.md (место в дереве проекта — по договорённости с преподавателем; в PDF имя задан явно).



При использовании ИИ: documents/lab9-numerical/2_criteria_tz.md, documents/lab9-numerical/2_criteria_plan.md, реализация строго по плану; после каждого завершённого шага — коммит и push.



Этап 3. Обработка ошибок (NaN и исключения)





NaN: обосновать и реализовать поведение, если функция невязки на произвольной итерации вернула NaN.



Исключения: обосновать и реализовать поведение, если вызываемая функция выбрасывает исключение.



При использовании ИИ: documents/lab9-numerical/3_exceptions_tz.md, documents/lab9-numerical/3_exceptions_plan.md, выполнение плана агентом.



Этап 4. Численная производная для Ньютона без дублирования метода





Придумать способ не писать вторую полную реализацию метода Ньютона (одна общая логика + подстановка аналитической или численной производной).



Добавить тест на корректность численного расчёта производной.



При использовании ИИ: documents/lab9-numerical/4_numerical_derivative_tz.md, documents/lab9-numerical/4_numerical_derivative_plan.md, реализация по плану; после каждой подзадачи — коммит и push.



Этап 5. Монотонная сходимость и диагностика по итерациям





Выдавать пользователю историю расчёта по итерациям (чтобы по ней можно было строить график, как в лекции).



В диагностике отразить, наблюдается ли монотонная сходимость.



Проанализировать влияние диагностики на производительность и типовой сценарий (бизнес-логика); обеспечить баланс удобство/скорость.



Ограничение из PDF: прямая работа с памятью в смысле низкоуровневых хаков — запрещена (использовать безопасные абстракции языка/стандартной библиотеки).



Проверить корректность формирования диагностики тестами.



При использовании ИИ: documents/lab9-numerical/5_analysis_tz.md как ТЗ; documents/lab9-numerical/5_analysis_plan.md — план реализации и тестирования; затем код в агентном режиме.



Каждый завершённый этап — отдельный коммит.



Этап 6. Полиморфизм и «чёрный ящик» расчётчика





Доработать код до обобщённого расчётчика с очевидным для пользователя интерфейсом; расчётчик — чёрный ящик.



Проанализировать: сейчас можно передать любые r и dr — ясно ли из API, что метод требует жёсткой связи r и dr? Предложить способ явно отразить связь на уровне интерфейса в духе LSP.



Продумать типовые сценарии и единую точку входа для запуска итерационного поиска корня с заданными параметрами.



Тесты: работа расчётчика на двух разных видах уравнений.



При использовании ИИ: documents/lab9-numerical/6_polymorphism_tz.md, documents/lab9-numerical/6_polymorphism_plan.md, затем код; коммит после каждой подзадачи.



Этап 7. Валидация через fixed_solvers





Независимая библиотека: fixed_solvers — клонировать к себе.



Зависимости для сборки fixed_solvers: Eigen и Boost.





Eigen (из PDF): клонировать eigen, в папке исходников создать каталог сборки msvc_cmake, команды из методички: cmake .. -G "Visual Studio 17 2022", cmake --build ., cmake --install . (из каталога eigen\msvc_cmake).



Собрать и установить fixed_solvers с CMake; для MSVC ориентироваться на fixed_solvers\msvc_cmake\readme.txt.



Численное исследование (research) оформить как тест (или тестовый модуль): сравнение своей бисекции/Ньютона с fixed_solvers; пример использования — fixed_solvers\testing\newton_solver.cpp.



Верификация: корень своим расчётчиком и эталонным; корни должны приблизительно совпадать; провести для всех уравнений, использованных в лабораторной ранее (линейное, квадратное, гиперболическое, гармоническое и прочие добавленные тестовые случаи).



Рекомендация из PDF: в дальнейшем предпочтительно использовать реализации из fixed_solvers.



При использовании ИИ: documents/lab9-numerical/7_validation_tz.md, documents/lab9-numerical/7_validation_plan.md, численная верификация в агентном режиме.
