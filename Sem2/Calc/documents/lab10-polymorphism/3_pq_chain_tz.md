Задача 3. Расчет PQ-цепочки на векторе указателей 3_pq_chain
Реализовать полиморфный проход по цепочке гидравлических объектов.
Требования к архитектуре:
ввести базовый интерфейс hydro_model_t ;
объявить в нем виртуальный метод solve_pq ;
все элементы цепочки наследовать от hydro_model_t ;
хранить цепочку в виде вектора указателей на базовый тип;
расчет цепочки выполнять единообразно: у каждого объекта вызывать solve_pq .
Пояснение по указателям и полиморфным вызовам
В этой лабораторной указатель нужен как механизм обращения к разным объектам через единый
базовый тип.
Когда переменная имеет тип указателя на базовый класс, а указывает на объект производного
класса, вызов виртуального метода выбирается по реальному типу объекта во время выполнения.
Это тот же принцип, который уже рассматривался на примере logger_base :
logger_base* logger = new file_logger();
logger->write_message("disk is almost full"); // вызовется
file_logger::write_message
delete logger; // обязательно: освобождаем память
Здесь принципиально важно, что объект создан через new , значит его обязательно нужно удалить
через delete . Если удалить забыть, возникнет утечка памяти: выделенная область памяти
останется занятой до завершения программы. При многократных созданиях объектов такие утечки
накапливаются, увеличивают потребление памяти и могут привести к нестабильной работе.
В расчетной части цепочки используется та же идея: в контейнере хранятся указатели на
hydro_model_t , а вызов solve_pq для каждого элемента выполняется полиморфно.
Проверка этапа:
тест на смешанную цепочку из нескольких типов объектов;
подтверждение, что нет ветвления по типам ( if/switch по конкретным классам) в
основном расчетном контуре.
отдельно продумать и зафиксировать решение для случая, когда в цепочке встретился
nullptr : можно ли вызывать методы по такому указателю, как это отразить в контракте
интерфейса и какое поведение выбрать в коде (исключение, пропуск элемента, и т.д.).
