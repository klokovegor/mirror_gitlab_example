# План реализации полиморфизма для расчёта гидравлической цепочки (лаб. 10)

Документ описывает **фактический порядок работ** и архитектурные решения, по которым в проекте `Sem2/Calc` введён полиморфный слой для `solve_pq` / `solve_qp` / `solve_pp` на уровне цепочки. Опирается на [3_polymorfizm_tz.md](3_polymorfizm_tz.md).

---

## 1. Цель

- Вынести общий контракт «звено цепочки» в базовый тип с виртуальными методами расчёта.
- Хранить последовательность звеньев как контейнер указателей на базовый тип и в основном расчётном контуре **не ветвиться** по конкретным классам (`if`/`switch` по типу C++-объекта).
- Сохранить уже реализованные калькуляторы элементов (`pipe_calculator_t`, `local_resistance_calculator_t`, и т.д.): звено — это **адаптер**, который проксирует вызовы в них.
- Постановку **PP для цепочки** реализовать как надстройку над **PQ**: внешний итеративный подбор расхода, внутри — тот же полиморфный проход `solve_pq`.

---

## 2. Этапы реализации

### Этап A. Базовый интерфейс и адаптеры звеньев

**Файлы:** `src/hydro_model.h`, `src/hydro_model.cpp`.

1. Ввести абстрактный класс `hydro_model_t`:
   - виртуальный деструктор `= default`;
   - чистые виртуальные `solve_pq()`, `solve_qp()`, `solve_pp()`.
2. Для каждого вида элемента из `type_of_obj_t` объявить финальный наследник:
   - `hydro_chain_pipe_element_t`
   - `hydro_chain_local_resistance_element_t`
   - `hydro_chain_pump_element_t`
   - `hydro_chain_pump_station_element_t`
3. В конструкторе адаптера зафиксировать ссылки на:
   - неизменяемые свойства задачи (`chain_task_properties_t`);
   - агрегированный результат (`chain_task_result_t`);
   - индексы в соответствующих векторах свойств;
   - **три скалярные ссылки**, общие для всей цепочки: `pressure_in`, `pressure_out`, `volume_flow` (те же поля, что у `chain_task_calculator_t`), чтобы при проходе менялись граничные величины между звеньями без дублирования состояния.
4. В каждом `solve_*` адаптера: собрать локальный калькулятор элемента, выставить граничные величины из ссылок, вызвать одноимённый `solve_*` калькулятора, записать результат в `chain_task_result_t` и обновить выходное давление или расход в соответствии с постановкой (как раньше делала монолитная логика по типу enum).

**Итог этапа:** полиморфный контракт «звено» и делегирование в существующий численный код элементов.

### Этап B. Цепочка как вектор `unique_ptr<hydro_model_t>`

**Файлы:** `src/hydraulic_chain.h`, `src/hydraulic_chain.cpp`.

1. В `chain_task_calculator_t` добавить поле  
   `std::vector<std::unique_ptr<hydro_model_t>> chain_elements`.
2. Вынести сборку вектора в локальную функцию `build_chain_elements(...)`:
   - один проход по `props.chain` (`std::vector<type_of_obj_t>`);
   - **`switch` только здесь** — по значению enum из входных данных, не по динамическому типу C++: для каждого `type_of_obj_t` создаётся `std::make_unique<...>(...)` нужного адаптера;
   - вести счётчики индексов по сегментам трубы, МС, насосу, НПС (как при заполнении параллельных массивов свойств).
3. В конструкторе `chain_task_calculator_t` инициализировать `chain_elements` вызовом `build_chain_elements`, передав ссылки на `pressure_in`, `pressure_out`, `volume_flow` и на `chain_task_result`.
4. Зафиксировать инвариант и проверки: в `chain_elements` **не должно быть** нулевых `unique_ptr`; при нарушении — явное исключение с индексом (см. раздел 4).

**Итог этапа:** цепочка хранится полиморфно; ветвление по «виду элемента в задаче» изолировано на этапе построения.

### Этап C. Полиморфные циклы `solve_pq` и `solve_qp`

**Файл:** `src/hydraulic_chain.cpp`.

1. **`solve_pq`:** прямой цикл по индексу `i`; для `i > 0` присвоить `pressure_in = pressure_out` (стык); затем `chain_elements[i]->solve_pq()` — без `switch` по классам.
2. **`solve_qp`:** обратный цикл (`rbegin`/`rend`); для всех звеньев кроме первого в обратном порядке перенос границы `pressure_out = pressure_in`; затем `(*it)->solve_qp()`.

**Итог этапа:** основной расчётный контур соответствует требованию ТЗ про отсутствие ветвления по типам объектов в цикле.

### Этап D. PP цепочки поверх PQ

**Файлы:** `src/hydraulic_chain.cpp`, `src/solve_pp_newton.h`, `src/solve_pp_newton.cpp`.

1. Зафиксировать граничные давления начала и конца цепочки (`pressure_start_boundary`, `pressure_end_target`).
2. Определить невязку по расходу: для пробного `q` выставить `pressure_in`, `volume_flow = q`, вызвать **`solve_pq()`** (полный полиморфный проход), сравнить полученный `pressure_out` с целевым — это `residual_at_q`.
3. Подобрать корень невязки внешним решателем (Ньютон с ограничениями / перебор начальных приближений при необходимости) — без дублирования логики расчёта отдельных элементов.
4. После успешного подбора ещё раз выполнить `solve_pq()` с найденным `volume_flow` и записать итог в `chain_task_result`.

**Итог этапа:** алгоритм PP цепочки не зависит от набора конкретных классов звеньев; расширение набора типов требует только нового наследника `hydro_model_t` и ветки в `build_chain_elements`, а не правок Ньютона.

### Этап E. Сборка проекта и тесты

**Файлы:** `CMakeLists.txt` (подключение `hydro_model.cpp`), `test/test_hydraulic_chain.cpp`.

1. Добавить `hydro_model.cpp` / `hydro_model.h` в цель `Tasks`.
2. Тесты:
   - смешанная цепочка в **PQ** (например труба + местное сопротивление) с сравнением с последовательным вызовом двух калькуляторов;
   - **PP** для смешанной цепочки (сходимость по выходному давлению, конечный расход);
   - по желанию — цепочка с **тремя и более разными типами** элементов для наглядной демонстрации полиморфного прохода;
   - комментарий в тестах про агрегированный `chain_task_result_t` (один блок результата на тип калькулятора; при нескольких элементах одного типа перезапись последнего расчёта).

---

## 3. Соответствие пунктам ТЗ (кратко)

| Пункт ТЗ | Реализация |
|----------|------------|
| `hydro_model_t`, `solve_pq` | Базовый класс + три виртуальных постановки в `hydro_model.h` |
| Наследники под все элементы | Четыре `hydro_chain_*_element_t` в `hydro_model.*` |
| Вектор указателей на базу | `chain_elements` в `hydraulic_chain.h` |
| Единообразный вызов в цикле | `solve_pq` / `solve_qp`: только `->solve_*()` |
| Нет ветвления в основном контуре | `switch` ограничен `build_chain_elements` |
| PP над PQ | `solve_pp` → `residual_at_q` → `solve_pq()` |
| `nullptr` в цепочке | См. раздел 4: проверка после фабрики + `ensure_chain_elements_non_null` в `solve_*` |

---

## 4. Контракт и `nullptr`

**Политика:** пустые звенья **не пропускаются** (иначе ломается согласование давлений на стыках). Нулевой указатель — нарушение контракта; вместо UB выбрасывается **`std::runtime_error`** с **индексом** проблемной позиции.

**Где реализовано** (`src/hydraulic_chain.cpp`, `src/hydraulic_chain.h`):

1. **После сборки вектора** — в конце `build_chain_elements` вызывается вспомогательная функция в анонимном namespace (`throw_if_nullptr_chain_element`): проход по всем элементам; если `!elements[i]`, сообщение вида  
   `chain_task_calculator_t: элемент цепочки с индексом N — нулевой указатель`.  
   Так ловятся ошибки будущих правок фабрики (случайный `push_back` пустого `unique_ptr`).

2. **Перед расчётом** — приватный метод `chain_task_calculator_t::ensure_chain_elements_non_null(const char* context)` с тем же правилом; префикс в сообщении — переданный `context` (например `chain_task_calculator_t::solve_pq`). Вызовы в начале **`solve_pq`**, **`solve_qp`** и сразу после валидации давлений в **`solve_pp`** (защита инварианта на входе в публичный расчётный контур).

3. **Документация в коде** — комментарий к полю `chain_elements` и к `ensure_chain_elements_non_null` в заголовке: ожидание отсутствия нулевых указателей, исключение при нарушении, пропуск не поддерживается.

**Примечание:** при текущей реализации фабрики только через `std::make_unique` нулевые указатели не создаются; проверки дают предсказуемое поведение при регрессиях и формально закрывают пункт ТЗ про продумывание случая `nullptr`. Отдельный unit-тест на вставку `nullptr` в приватный вектор не обязателен.

---

## 5. Зависимости и замечания

- Численные решатели (в т.ч. `fixed_solvers`) подключаются к калькуляторам элементов и к модулю PP по расходу; полиморфный слой цепочки от них не зависит напрямую.
- Заголовок `hydraulic_chain.h` содержит forward-declaration `hydro_model_t`, чтобы разорвать циклические включения с `hydro_model.h` (полное определение подключается в `.cpp`).

---

## 6. Проверка готовности

- Сборка библиотеки `Tasks` и исполняемого `Tests`.
- Прогон тестов: в первую очередь сценарии из `test_hydraulic_chain.cpp` (PQ/PP смешанной цепочки) и регрессия по `test_pipe_oil.cpp` / элементам.

Этого достаточно, чтобы зафиксировать **план, по которому реализован полиморфизм**, и воспроизвести или продолжить разработку по тем же шагам.
